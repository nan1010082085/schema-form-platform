智能体设计

三种专家 Agent 各司其职，Router 仅在 standalone 模式下使用，编辑器/流程场景按 source 直接分发

◉ Router Agent

职责：意图识别，将 standalone 请求分发到对应专家

触发条件

仅在 source === 'standalone' 时激活

实现

调用 DeepSeek 分类意图，返回 { target: 'editor' | 'flow' }，失败时降级为关键词匹配

LLM 意图分类 关键词降级

◉ Editor Agent

职责：生成/修改 Widget Schema（表单、列表页、业务页面）

领域知识

47 种 Widget（8 组：容器 2、布局 8、表单 18、静态 2、操作 2、表格 3、业务 3、图表 9）

输出格式

<think> + <answer> + <tip> + <schema>

Widget 生成 工具调用 Schema 校验

◉ Flow Agent

职责：生成/修改流程图（FlowGraph）

领域知识

12 种 BPMN 节点类型、审批模式（会签/或签）、网关条件、表达式语法

输出格式

<think> + <answer> + <tip> + <schema>（flow_update）

流程生成 工具调用 Flow 校验

工具调用循环（核心机制）

Editor Agent 和 Flow Agent 共享相同的工具调用循环模式：while(true) + LLM streaming + tool_calls 累积 + 执行 + 写回

// Agent 工具调用循环（agentBase.ts 共享基础设施） while (true) { // 1. LLM streaming 请求（带 tools 定义） const stream = await openai.chat.completions.create({ model: 'deepseek-chat', messages, tools: EDITOR_TOOLS, // 或 FLOW_TOOLS tool_choice: 'auto', stream: true, }) // 2. 遍历 stream chunk，累积 tool_calls for await (const chunk of stream) { if (delta.tool_calls) { /* 累积到 toolCallMap */ } else { /* 累积 content，提取 thinking 流式事件 */ } } // 3. 无 tool_calls → 跳出循环 if (toolCallMap.size === 0) break // 4. 超过 5 轮 → 报错跳出 toolRound++ if (toolRound > MAX_TOOL_ROUNDS) break // 5. 执行工具 → 结果写入 messages const toolCalls = parseToolCalls(toolCallMap) yield { type: 'tool_call', toolCalls } // 前端展示 for (const tc of toolCalls) { const result = await executeToolWithTimeout(tc) messages.push({ role: 'tool', tool_call_id: tc.id, content: result }) } yield { type: 'tool_call', toolResults } // 6. 重置状态 → 下一轮 LLM（带工具结果） fullContent = '' } // 循环结束后 → parseStructuredOutput(fullContent) // → yield text / tip / schema / flow

基础设施 agentBase.ts

两个 Agent 共享的底层能力：

• getClient() — OpenAI 单例（baseURL: api.deepseek.com）

• buildMessages() — 历史消息 + system prompt 构建

• parseStructuredOutput() — XML 标签解析

• processThinkingDelta() — think 标签流式提取

• withRetry() — 指数退避重试（429/5xx）

• executeToolWithTimeout() — 30s 超时执行

结构化输出 XML 标签

工具调用循环结束后，从最终 content 中解析：

<!-- Editor Agent 输出 --> <think>分析用户需求...</think> <answer>已为你生成表单...</answer> <tip>建议添加必填校验...</tip> <schema>{ type: "schema_update", widgets: [...] }</schema>

容错机制

• 工具超时：每个工具 30s 限时

• 轮次上限：最多 5 轮工具调用

• 重试策略：指数退避，最多 3 次

• Router 降级：LLM 失败时用关键词匹配

• 输出容错：无 answer 标签时回退到去除 think 后的纯文本

• SSE 心跳：15s 间隔保持连接

SSE 事件流

服务端通过 Server-Sent Events 流式推送 7 种事件类型 + 1 种完成信号

type含义数据结构
thinkingLLM 思考过程流式输出{ content: string }
text正式回答文本{ content: string }
tip使用建议 / 优化提示{ content: string }
schema生成的 Widget Schema{ payload: Widget[], description: string }
flow生成的 FlowGraph{ payload: FlowGraph, description: string }
tool_call工具调用（calling / result 两阶段）{ phase, tools: [...] }
done流结束，携带 conversationId{ conversationId: string }
error错误信息{ content: string }

SSE 流示例（带工具调用）

event: thinking
data: {"content": "用户需要一个请假表单..."}
 
event: tool_call
data: {"phase": "calling", "tools": [{"name": "get_widget_catalogue", ...}]}
 
event: tool_call
data: {"phase": "result", "tools": [{"name": "get_widget_catalogue", "result": {...}}]}
 
event: text
data: {"content": "已为你生成请假申请表单..."}
 
event: schema
data: {"payload": [...], "description": "请假申请表单"}
 
event: done
data: {"conversationId": "conv_xxx"}

SSE 客户端实现

前端使用 fetch + ReadableStream（非 EventSource），因为需要 POST 请求体。

// ai-app SSE 客户端 const response = await fetch('/api/ai/chat', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ message, context }), }) const reader = response.body!.getReader() const decoder = new TextDecoder() while (true) { const { done, value } = await reader.read() if (done) break const text = decoder.decode(value) // 解析 "data: {...}\n\n" 格式 parseSSELines(text, handleEvent) }

选择 fetch 而非 EventSource 的原因：EventSource 仅支持 GET，无法发送 POST 请求体。

前端 ai-app

独立 Vue 3 + Pinia 微应用，两种入口模式，通过 bridge 与宿主通信

两种入口模式

AiChatView（完整页面）

三栏布局：对话列表 + 聊天面板 + 预览面板

• 左侧：AiConversationList 对话历史列表
• 中间：AiChatPanel 消息流 + 输入框
• 右侧：AiPreviewPanel Schema/Flow 预览
• 入口：main.ts → router.ts → /ai

AiSidebarView（抽屉模式）

400px 单栏，嵌入 Editor / Flow 宿主

• 仅 AiChatPanel 聊天面板
• 预览通过 bridge 推送到宿主
• 入口：main-sidebar.ts → 抽屉组件
• 宿主通过 micro-app 嵌入

AiChatPanel.vue

对话主界面，管理消息列表、输入框、SSE 流式接收。展示 thinking 折叠区、tool_call 进度、消息渲染。

AiMessage.vue

单条消息渲染，支持纯文本、thinking 折叠区、tip 提示。

SchemaCard.vue / FlowCard.vue

生成物卡片组件，展示 Schema 预览或 FlowGraph 预览，支持发布操作。

AiPreviewPanel.vue

右侧面板，实时渲染生成的 Schema 或 FlowGraph，支持交互预览。

AiConversationList.vue

对话历史列表，支持切换、删除对话，显示对话来源和 Agent 标识。

bridge.ts

宿主通信桥接，支持 micro-app 模式（dispatch/getData）和 standalone 模式（postMessage）双通道。

嵌入方式与通信协议

ai-app 通过 Socket + bridge 双通道与宿主通信，支持 micro-app 和 postMessage 两种模式

Editor 中嵌入

<!-- Editor 侧边栏 --> <micro-app name="ai-app" :url="AI_APP_URL" :data="{ source: 'editor', schemaId }" />

Flow 中嵌入

<!-- Flow 属性面板 --> <micro-app name="ai-app" :url="AI_APP_URL" :data="{ source: 'flow', flowId, nodeId }" />

通信协议（bridge.ts）

方向事件说明
ai-app → 宿主ai:preview-schema推送生成的 Schema 到编辑器预览
ai-app → 宿主ai:preview-flow推送生成的 FlowGraph 到流程设计器预览
ai-app → 宿主ai:published通知发布完成及绑定结果
ai-app → 宿主ai:open-in-editor请求宿主在编辑器中打开生成物
宿主 → ai-appai:set-context传入当前上下文（source, IDs）
宿主 → ai-appai:current-schema传入当前编辑器 Schema 供 AI 参考

bridge 双通道实现

micro-app 模式

发送：window.microApp.dispatch()
接收：microApp.addEventListener('datachange')

standalone 模式

发送：window.parent.postMessage()
接收：window.addEventListener('message')
