智能体设计

三种专家 Agent 各司其职，Router 负责分发与编排

◉ Router Agent

职责：意图识别，将用户请求分发到对应专家，或协调多专家协作

路由规则

涉及表单、列表、页面、UI、组件 → Editor Agent

涉及流程、审批、节点、分支、条件 → Flow Agent

跨域需求（节点绑定表单） → 协作模式

意图分类 任务拆分 结果合并

◉ Editor Agent

职责：生成/修改 Widget Schema（表单、列表页、业务页面）

领域知识

35+ Widget 类型系统、布局容器嵌套、16 种事件动作、联动规则、校验规则

Widget 生成 布局计算 Schema 校验

◉ Flow Agent

职责：生成/修改流程图（FlowGraph）

领域知识

12 种 BPMN 节点类型、审批模式（会签/或签）、网关条件、表达式语法

流程生成 节点配置 表达式生成

AI 基础设施：Memory / RAG / 向量库

Agent 的记忆体系、知识检索策略与向量库选型

Memory 体系

LangGraph 原生支持多层记忆，无需额外框架

短期记忆

用途：当前对话上下文、多轮消息

实现：LangGraph State 的 messages[]

生命周期：单次会话

工作记忆

用途：当前正在构建的 Schema / Flow

实现：LangGraph State 的 currentSchema / currentFlow

生命周期：单次会话

长期记忆

用途：跨会话的用户偏好、历史对话

实现：MongoDB AIConversation 集合

生命周期：持久化

RAG 策略

核心问题：Agent 如何获取项目领域知识（Widget 类型、节点配置、Schema 结构）？

Phase 1：纯 Prompt + 基础向量存储

将领域知识直接注入系统提示词，同时引入 @langchain/mongodb 做对话历史持久化。Widget 类型是有限集（35 种），系统提示词完全放得下。

// Editor Agent 系统提示词包含： const editorSystemPrompt = ` 你是表单/页面生成专家。 以下是可用的 Widget 类型： ## 基础组件 - input: 文本输入框 - select: 下拉选择 - number: 数字输入 - date: 日期选择 - ... ## 布局组件 - form: 表单容器 - card: 卡片容器 - row-col: 栅格布局 - tabs: 选项卡 ## 规则 1. 基础组件只能嵌套在布局组件内 2. 每个组件必须有 position 3. 布局坐标不能重叠 `
延迟最低 实现最简 复用 MongoDB 实例

Phase 2+：RAG（按需扩展）

当以下条件满足时引入 RAG：

• 用户自定义 Widget / 模板数量增长到 Prompt 放不下

• 需要检索历史生成的优秀案例作为参考

• 需要理解项目特定的业务术语和字段命名习惯

动态知识更新 精准检索 案例复用

向量库选型

Phase 1 即引入 @langchain/mongodb，使用 MongoDB Atlas Vector Search 作为向量存储

已选定 MongoDB Atlas Vector Search

通过 @langchain/mongodb 包提供两个核心能力：

MongoDBChatMessageHistory

对话历史持久化，复用现有 MongoDB 实例

Phase 1 使用
MongoDBAtlasVectorSearch

向量存储与语义检索，支持 RAG

Phase 2+ 使用

• 零额外基础设施（复用现有 MongoDB Atlas）
• 对话数据和向量数据在同一数据库，查询效率高
• Phase 1 先用对话持久化，Phase 2 开启向量检索，无需新增依赖

对话流程

一次典型的 AI 交互：从自然语言描述到 Schema 生成与发布

U
User
帮我做一个请假申请表单，需要请假类型、起止日期、事由、附件
R
Router Agent → 识别为编辑器任务，分发到 Editor Agent
路由决策：target=editor, task="生成请假申请表单"
E
Editor Agent
已为你生成请假申请表单，包含以下字段：
• 请假类型（Select：事假/年假/病假/婚假/产假）
• 开始日期（DatePicker）
• 结束日期（DatePicker）
• 请假事由（Textarea，必填）
• 附件上传（Upload，可选）

Widget[] Schema 已生成，请在右侧预览面板查看。

User
再加一个紧急联系人字段
E
Editor Agent
已在表单末尾添加「紧急联系人」Input 字段（必填），Schema 已更新。
U
User
确认，发布并绑定到当前节点
AI
System
表单已发布（publishId: pf_abc123），已绑定到流程节点「请假申请」。

协作模式

当用户需求跨越编辑器和流程两个领域时，Router 拆分任务并协调多 Agent 协作

示例：创建请假审批流程

用户请求："创建一个请假审批流程，包含请假表单"

Router 拆分任务
→
Editor Agent: 生成请假表单
→
Flow Agent: 生成审批流程
→
合并结果 & 发布
// LangGraph 协作子图 const collaborationGraph = new StateGraph(stateAnnotation) .addNode("router", routerNode) .addNode("editor", editorAgentNode) .addNode("flow", flowAgentNode) .addNode("merge", mergeResultsNode) .addEdge("router", "editor") .addEdge("editor", "flow") // Editor 先生成表单 .addEdge("flow", "merge") // Flow 用 publishId 生成流程 .addEdge("merge", "__end__")

API 设计

三个核心接口，SSE 流式对话 + 发布绑定 + 对话管理

POST /api/ai/chat

对话接口，SSE 流式响应。支持多轮对话，自动路由到对应 Agent。

// Request { "conversationId": "conv_xxx", // 首次不传 "message": "帮我做一个请假表单", "context": { "source": "flow", "flowId": "flow_xxx", "nodeId": "node_xxx" } }

SSE 响应事件流

服务端通过 Server-Sent Events 流式推送生成结果

event: text
data: {"content": "已为你生成..."}
 
event: schema
data: {"payload": [...], "description": "表单 Schema"}
 
event: done
data: {"conversationId": "conv_xxx"}

POST /api/ai/publish

发布生成物并绑定到目标（流程节点）

// Request { "conversationId": "conv_xxx", "type": "schema", "payload": [ /* Widget[] */ ], "target": { "type": "flow_node", "flowId": "flow_xxx", "nodeId": "node_xxx" } } // Response { "id": "schema_xxx", "publishId": "pf_abc123", "boundTo": { "flowId": "...", "nodeId": "..." } }

GET/DELETE /api/ai/conversations/:id

对话历史管理

// GET Response { "id": "conv_xxx", "source": "flow", "messages": [ { "role": "user", "content": "..." }, { "role": "assistant", "content": "...", "schema": [/* Widget[] */] } ], "activeAgent": "editor", "createdAt": "2026-05-29T..." }

对话状态模型

LangGraph StateGraph 的核心状态定义

interface AIConversationState { // 对话历史 messages: AIMessage[] // 当前活跃的 Agent activeAgent: 'router' | 'editor' | 'flow' // 业务上下文 context: { source: 'editor' | 'flow' | 'standalone' schemaId?: string // 编辑器场景：当前 Schema ID flowId?: string // 流程场景：当前流程 ID nodeId?: string // 流程场景：当前节点 ID currentSchema?: Widget[] // 当前生成的 Schema currentFlow?: FlowGraph // 当前生成的 Flow turnCount: number // 对话轮次 } // 待确认操作 pendingAction?: { type: 'preview' | 'publish' | 'modify' payload: Widget[] | FlowGraph description: string } }

嵌入方式

ai-app 通过 microApp 嵌入宿主，通过 postMessage 双向通信

Editor 中嵌入

<!-- Editor 侧边栏 --> <micro-app name="ai-app" :url="AI_APP_URL" :data="{ source: 'editor', schemaId }" />

Flow 中嵌入

<!-- Flow 属性面板 --> <micro-app name="ai-app" :url="AI_APP_URL" :data="{ source: 'flow', flowId, nodeId }" />

通信协议

方向事件说明
ai-app → 宿主ai:preview-schema推送生成的 Schema 到编辑器预览
ai-app → 宿主ai:preview-flow推送生成的 FlowGraph 到流程设计器预览
ai-app → 宿主ai:published通知发布完成及绑定结果
宿主 → ai-appai:set-context传入当前上下文（source, IDs）
宿主 → ai-appai:current-schema传入当前编辑器 Schema 供 AI 参考

实施阶段

分四阶段逐步落地，每阶段可独立验证

Phase 1
最小闭环
Server：Router Agent + Editor Agent，单轮生成 Widget Schema
ai-app：对话 UI + Schema 预览 + 发布
验证：描述 → 生成表单 → 预览 → 发布到 Editor
Phase 2
多轮对话
Editor Agent 支持上下文感知的修改（"再加一个字段"）
对话持久化（MongoDB）
SSE 流式响应
Phase 3
Flow Agent
Flow Agent 实现，支持流程图生成
协作模式：Editor + Flow 联合生成
发布绑定到 Flow 节点
Phase 4
能力增强
Agent Tools 完善（模板库、校验、布局计算）
对话历史恢复
Schema 校验失败后自动修正
